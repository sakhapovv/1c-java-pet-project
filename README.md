# 1C + Java интернет-магазин

Pet-проект интернет-магазина с интеграцией `1С:Предприятие 8` и Java Spring Boot.

Java-приложение отвечает за сайт: каталог товаров, корзину, регистрацию, оформление заказа и личный кабинет. 1С выступает учетной системой и источником данных по товарам, остаткам, заказам, оплатам и отгрузкам.

## Стек

- Java
- Spring Boot
- Spring Web
- Spring Data JPA
- Spring Security
- Thymeleaf
- PostgreSQL
- Docker
- 1С HTTP-сервисы
- JSON API

## Интеграция с 1С

### Получение товаров из 1С

Java получает каталог из 1С:

```text
GET /InfoBase/hs/site/products
```

Пример ответа:

```json
[
  {
    "id": "200169cf-496b-11f1-8c0a-d8bbc16ddaff",
    "name": "Ноутбук Acer",
    "article": "ACER-001",
    "category": "Ноутбуки",
    "active": true,
    "price": 80000,
    "stock": 5,
    "imageUrl": "/InfoBase/hs/site/products/200169cf-496b-11f1-8c0a-d8bbc16ddaff/image"
  }
]
```

### Создание заказа в 1С

После оформления заказа Java отправляет его в 1С:

```text
POST /InfoBase/hs/site/orders
```

Пример запроса:

```json
{
  "orderId": "8c032c8e-4d7b-4a52-9d50-7ff3d88bdb47",
  "customer": {
    "siteId": "user-external-id",
    "name": "Alex",
    "phone": "123",
    "email": "alex@mail.com"
  },
  "items": [
    {
      "productId": "200169cf-496b-11f1-8c0a-d8bbc16ddaff",
      "quantity": 1
    }
  ]
}
```

### Обновление статуса заказа из 1С

При изменении статуса заказа 1С отправляет обновление в Java:

```text
POST /api/orders/status
```

Пример запроса:

```json
{
  "orderId": "8c032c8e-4d7b-4a52-9d50-7ff3d88bdb47",
  "status": "Оплачен",
  "order": "Заказ клиента 000000001"
}
```

Соответствие статусов:

```text
Новый     -> SENT_TO_1C
Оплачен   -> PAID
Отгружен  -> SHIPPED
Отменен   -> CANCELLED
```

## Повторная отправка статусов заказов

Основной механизм обновления статуса работает сразу после изменения заказа в 1С: 1С вызывает Java endpoint `/api/orders/status`. Если Java-приложение временно недоступно, статус не теряется, а записывается в очередь повторной отправки.

В 1С создан регистр сведений:

```text
ОчередьСтатусовЗаказовСайта
```

Структура регистра:

```text
Измерения:
- Заказ: ДокументСсылка.ЗаказКлиента

Ресурсы:
- Статус: ПеречислениеСсылка.СтатусЗаказа

Реквизиты:
- ИдентификаторЗаказаСайта
- ДатаСоздания
- ДатаПоследнейПопытки
- КоличествоПопыток
- Отправлено
- ТекстОшибки
```

Логика работы:

1. 1С меняет статус заказа.
2. 1С пытается отправить статус в Java.
3. Если Java ответила успешно, в регистре фиксируется `Отправлено = Истина`.
4. Если Java недоступна или вернула ошибку, в регистре фиксируется `Отправлено = Ложь`, увеличивается `КоличествоПопыток`, сохраняется `ТекстОшибки`.
5. Регламентное задание 1С каждые 5 минут повторяет отправку записей, где `Отправлено = Ложь`.
6. После успешной повторной отправки запись помечается как `Отправлено = Истина`.

Регламентное задание вызывает процедуру общего модуля:

```bsl
ИнтеграцияСайтаСервер.ПовторитьОтправкуСтатусовЗаказовНаСайт()
```

Ограничение повторов:

```text
КоличествоПопыток < 10
```

Это защищает систему от бесконечной отправки при неправильных настройках интеграции.

### Проверка повторной отправки

1. Остановить Java-приложение.
2. Изменить статус заказа в 1С.
3. Проверить регистр `ОчередьСтатусовЗаказовСайта`: должна появиться запись с `Отправлено = Ложь`.
4. Запустить Java-приложение.
5. Дождаться выполнения регламентного задания.
6. Проверить, что запись стала `Отправлено = Истина`, а статус заказа обновился на сайте.
